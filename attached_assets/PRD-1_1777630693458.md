## 1. PROJECT OVERVIEW

### 1.1 Product Name
Transsion Core Federation Bot (TC-Bot)

### 1.2 Purpose
Bot Telegram production-grade untuk Transsion Core Federation (TCF) — federasi komunitas independen yang mengelola grup-grup Telegram perangkat Infinix, Tecno, dan Itel. Bot ini memusatkan seluruh sistem moderasi, penegakan ban lintas grup secara otomatis, hierarki admin, koneksi grup, dan sistem banding (appeal) dari satu sistem terpusat.

### 1.3 Visi
Menggantikan moderasi manual per-grup yang terfragmentasi dengan sistem federasi terpadu dan sepenuhnya otomatis — satu perintah ban langsung berlaku di seluruh grup yang terhubung, banding diproses secara transparan, dan setiap tindakan dapat diaudit melalui kanal log khusus.

### 1.4 Technology Stack

| Layer | Teknologi |
|---|---|
| Language | Python 3.11+ |
| Telegram Library | python-telegram-bot 22.5 (async, Application.builder()) dengan JobQueue |
| Database | MongoDB (database name dari ENV `DB_NAME`) via motor async driver |
| Keep-alive Server | Flask (minimal, port dari ENV `PORT`, daemon thread) |
| Environment Config | python-dotenv (file: `config.env`) |
| Runtime | `python -m tgbot` |
| Platform Support | Linux, macOS, Windows, Android/Termux, Replit, Docker |

### 1.5 Constraints dan Non-Negotiables
- Tidak ada emoji di mana pun: pesan, caption, tombol, komentar, atau dokumentasi.
- Tidak ada JavaScript, TypeScript, Node.js, package.json, atau web framework selain Flask keep-alive.
- File Dockerfile, docker-compose.yml, dan `.github/workflows/*` adalah pre-existing — jangan disentuh.
- Semua pesan menggunakan HTML parse mode.
- Semua timestamp UTC, format: `DD-MM-YYYY | HH:MM`.
- Setiap pesan log ke LOG_CHANNEL wajib mengandung baris branding tepat: `𝘛𝘊𝘍 - 𝘛𝘳𝘢𝘯𝘴𝘴𝘪𝘰𝘯 𝘊𝘰𝘳𝘦 𝘍𝘦𝘥𝘦𝘳𝘢𝘵𝘪𝘰𝘯`
- Tidak ada satu file Python pun yang boleh melebihi 600 baris. Split jika perlu.
- Kredensial tidak pernah di-hardcode — selalu dari environment variables.
- Copyright block di paling atas setiap file Python:
  ```python
  # © Copyright 2024 - 2026 Transsion Core
  # © Copyright 2024 - 2026 Dizzy
  # © Copyright 2026 Aveum Apps
  ```
- Semua pesan bot menggunakan tone friendly-formal: hangat tapi tidak casual, sopan tapi tidak kaku.

### 2.2 Cara Bot Membaca config.env

Semua environment variable dibaca di `tgbot/__init__.py` menggunakan `python-dotenv`:

```python
from dotenv import load_dotenv
import os, json

load_dotenv("config.env")

BOT_TOKEN       = os.getenv("BOT_TOKEN")
OWNER_ID        = int(os.getenv("OWNER", "0"))
MONGODB_URI     = os.getenv("MONGODB_STRINGS")
DB_NAME         = os.getenv("DB_NAME", "tcbot")
COMMUNITY_NAME  = os.getenv("COMMUNITYS_NAME", "Transsion Core Federation")
PREFIXES        = json.loads(os.getenv("PREFIXES", '["/", "!", "."]'))
PORT_CONFIG     = os.getenv("PORT", "8080")
```

### 2.3 Logika Penentuan PORT

Di `tgbot/alive.py`:
- Jika `PORT_CONFIG` adalah angka (misal `"8080"`): gunakan langsung.
- Jika `PORT_CONFIG == "auto"`: cari port kosong secara acak menggunakan `socket` binding loop dalam range 1024–65535.

### 2.4 Logika Penentuan LOGS_ERROR

Di `tgbot/__init__.py`, saat startup:
1. Parse nilai `LOGS_ERROR`.
2. Jika string berformat `"-100xxx/yyy"`: gunakan chat_id dan thread_id tersebut.
3. Jika `"auto"`: cari topik dengan nama `"Logs Error"` di `MAIN_GROUP`. Jika tidak ada, set `LOGS_ERROR_DEST = None` (kirim ke PM owner).
4. Jika `"MAIN_GROUP"`: seperti `auto`, tetapi jika tidak ada buat topik baru bernama `"Logs Error"`.
5. Jika kosong: set `LOGS_ERROR_DEST = None`.

### 2.5 Logika Penentuan APPEALS

Di `tgbot/__init__.py`, saat startup:
1. Jika string berformat `"-100xxx/yyy"`: gunakan langsung.
2. Jika `"MAIN_GROUP"`: cari topik bernama `"Appeal"` di `MAIN_GROUP`. Jika tidak ada, buat topik baru bernama `"Appeal"`.

```
{
  chat_id:    int,       # Telegram chat ID (unique index)
  title:      str,       # Judul grup
  added_by:   int,       # user_id pemilik grup yang melakukan koneksi

  is_active:  bool       # false = diskoneksi atau bot dikeluarkan
}
```

### 4.2 Collection: `tc_owners`

```
{
  user_id:       int,
  promoted_by:   int,
  promoted_date: datetime

```
Index: `user_id` (unique)

### 5.1 Keputusan Arsitektur Kunci

**`workflows/` sub-package**: Semua ConversationHandler state machine diisolasi di sini. File `handlers/` tetap tipis — hanya mendaftarkan handler, tidak mengimplementasikan logic.

**`modules/helper/` sebagai shared utility layer**: Keyboards, message parsing, log building, link construction, HTML formatting. Tidak ada business logic, hanya builders dan transformers.

**`database/` adalah murni repository layer**: Tidak ada business logic. Setiap file hanya mengekspos fungsi async CRUD.

**`handlers/` adalah murni Telegram update layer**: Handler menerima update, memanggil modules, mengembalikan respons.

**`utils/` benar-benar generik**: Permission checks, auth checks, time formatting, prefix dispatch, logging.

**`kicking.py`** hadir di tiga lapisan:
- `database/kicks.py` — repository koleksi `kicks`
- `modules/kicking.py` — business logic kick
- `handlers/kicking.py` — Telegram update handler untuk /kick dan /kickme
- `modules/helper/workflows/kicking_flow.py` — flow validasi dan dispatch log

## 7. MODEL OTORISASI

### 7.1 Peran
- **TC Owner**: Disimpan di `tc_owners`. Tepat satu dokumen. Memiliki semua izin.
- **TC Admin**: Disimpan di `tc_admins`. Banyak dokumen. Memiliki hampir semua izin kecuali demosi dan transfer kepemilikan.
- **Group Owner**: Status "creator" Telegram. Hanya mengontrol koneksi grup miliknya sendiri.
- **Regular User**: Dapat mengecek status ban sendiri dan mengajukan appeal.

### 7.2 Aturan
- Untuk /tcban dan /tcunban: otorisasi dicek terhadap DB saja — tidak perlu menjadi Telegram admin di grup tersebut.
- Proteksi ban: tidak dapat mem-ban user yang ada di tc_owners atau tc_admins.
- Proteksi kick: tidak dapat men-kick user yang ada di tc_owners atau tc_admins.
- Self-target prevention: pesan spesifik per perintah (lihat Bagian 11).
- Promosi oleh TC Admin membuat pending request; TC Owner mempromosikan langsung.
- Hanya TC Owner yang dapat mendemosi, mentransfer, atau approve/reject promotion requests.
- Authorization guards diimplementasikan sebagai decorators di `modules/helper/decorators.py`.

## 8. SPESIFIKASI FITUR LENGKAP

### FITUR 1: Koneksi Grup Saat Bot Ditambahkan

**Trigger**: Bot ditambahkan ke grup.

**Alur**:

**Step 1**: Bot mengirim:
- Teks: "Do you want this community to join the Transsion Core Federation?"
- Inline keyboard (satu baris): "Join Transsion Core" (`tc_join`) | "Cancel" (`tc_cancel`)

**Step 2**: Hanya group owner (status "creator") yang boleh berinteraksi. User lain: alert "Only the group owner can decide."

**Step 3 — Owner klik "Join Transsion Core"**:
- Cek permission bot: `can_delete_messages`, `can_restrict_members`, `can_invite_users`.
- Jika kurang: edit pesan instruksi, simpan ke `pending_joins`, monitor `my_chat_member`.
- Jika cukup: upsert `federated_groups`, seed owner jika perlu, edit pesan sukses, log, trigger member cache init.

**Step 4 — Owner klik "Cancel"**: Edit pesan, leave group, log.

**Step 5 — Monitoring pending_joins**: Saat bot dipromosikan dan permission cukup, selesaikan join otomatis.

### FITUR 39: Kick (/kick, /tckkick, /kickout)

**File utama**: `handlers/kicking.py`, `modules/kicking.py`, `modules/helper/workflows/kicking_flow.py`, `database/kicks.py`

**Otorisasi**: TC Owner/Admin.

**Scope**: Hanya di grup yang terhubung (federated group). Tidak berlaku di PM atau MAIN_GROUP/EXEC_GROUP kecuali grup tersebut juga terhubung.

**Perbedaan mendasar dengan ban**:
- Kick = user dikeluarkan dari satu grup saja, dapat bergabung kembali kapan saja.
- Ban = user diblokir di seluruh grup terhubung, tidak dapat bergabung kembali tanpa proses unban/appeal.
- Kick tidak memerlukan proof upload.
- Kick tidak memiliki state aktif/nonaktif — hanya audit log.

**Usage**: `/kick <target> [alasan]`

Alasan bersifat opsional tetapi sangat dianjurkan untuk keperluan audit.

**Alur Kick**:

Step 1 — Entry:
- Validasi otorisasi: sender harus TC Owner atau TC Admin.
- Resolve target.
- Validasi proteksi: jika target adalah TC Owner atau TC Admin, reply "Cannot kick a Transsion Core Admin or Owner." dan stop.
- Validasi self-kick: jika target adalah sender, reply "You cannot kick yourself." dan stop.
- Validasi scope: jika dijalankan bukan di grup terhubung, reply "This command can only be used in a connected group." dan stop.

Step 2 — Eksekusi:
- Panggil `ban_chat_member(chat_id, target_id)` dengan `until_date=0` (ban instan lalu langsung unban = efek kick Telegram).
  - Alternatif yang lebih clean: panggil `unban_chat_member` segera setelah `ban_chat_member` sehingga user dikeluarkan tetapi tidak masuk daftar ban permanen di grup tersebut.
- Tangkap exception. Jika bot tidak memiliki `can_restrict_members`: reply "I do not have permission to kick members in this group." dan stop.

Step 3 — Record dan Log:
- Simpan ke koleksi `kicks`: `{ kick_id, kicked_user_id, chat_id, reason, admin_user_id, timestamp }`.
- Reply: "User \<target_id\> has been kicked from this group."
- Log ke LOG_CHANNEL (lihat template di bawah).

**Template Log — Kick (LOG_CHANNEL)**:
```
Member Kicked
𝘛𝘊𝘍 - 𝘛𝘳𝘢𝘯𝘴𝘴𝘪𝘰𝘯 𝘊𝘰𝘳𝘦 𝘍𝘦𝘥𝘦𝘳𝘢𝘵𝘪𝘰𝘯

Admin: <a href="tg://user?id=<admin_id>">Admin First Name</a>

User: <a href="tg://user?id=<kicked_user_id>">User First Name</a>
User ID: <kicked_user_id>

Group: <group_display>
Group ID: <chat_id>

[Reason: <reason>]  (jika diberikan)

Date: <DD-MM-YYYY> | <HH:MM>
```

`group_display`: link yang dapat diklik jika grup publik, jika tidak hanya nama grup.

**Catatan implementasi `kicking_flow.py`**:
- Flow ini tidak menggunakan ConversationHandler (tidak ada state machine).
- File ini hanya bertugas: (1) menerima data kick yang sudah divalidasi dari `modules/kicking.py`, (2) memanggil `database/kicks.py` untuk menyimpan record, (3) memanggil `modules/helper/parse_logmsg.py` untuk membangun log message, (4) memanggil `utils/logger.py` untuk mengirimnya ke LOG_CHANNEL.
- Pemisahan ini menjaga `handlers/kicking.py` tetap tipis dan `modules/kicking.py` terbebas dari detail logging.

```html

Usage: /kick <target> [reason]
Who can use: Transsion Core Owner and Admins.
Where: Connected groups only.
Note: Kick removes a user from the current group only. The user can rejoin at any time. This is not a federation-wide action. For a permanent federation-wide removal, use /tcban instead.
```

## 18. CATATAN IMPLEMENTASI UNTUK AI AGENT

- `utils/prefixes.py` menghasilkan registrasi CommandHandler untuk `/cmd`, `.cmd`, dan `!cmd` untuk setiap perintah, termasuk `/kick` dan alias-nya.
- `enforce_ban()` dan `enforce_unban()` adalah single entry point untuk cross-group enforcement. Kick TIDAK menggunakan fungsi ini.
- `modules/helper/parse_logmsg.py` adalah single source of truth untuk semua log builders, termasuk template log kick baru.
- `modules/helper/keyboards.py` adalah single source of truth untuk semua `InlineKeyboardMarkup`.
- `database/kicks.py` hanya menyimpan record audit — tidak ada field `is_active`.
- `handlers/kicking.py` hanya bertanggung jawab menerima update Telegram dan memanggil `modules/kicking.py`. Tidak ada logic di handler.
- `modules/kicking.py` menangani validasi business rule kick (otorisasi, proteksi, scope) dan memanggil Telegram API serta `kicking_flow.py` untuk log.
- `modules/helper/workflows/kicking_flow.py` bukan ConversationHandler. Hanya sebuah async function yang menerima data kick terstruktur, menyimpan ke DB, dan mengirim log.
- Variabel nama "affiliate" dan variannya diganti seluruhnya menjadi "connection" atau "connected" di seluruh kode, pesan, komentar, dan dokumentasi.
- File `agents/` wajib selalu sinkron dengan implementasi aktual.
- Direktori `tests/` menggunakan TDD. File test untuk `kicking.py` wajib dibuat di `tests/`.
- Jangan generate atau modifikasi: `Dockerfile`, `docker-compose.yml`, `.github/workflows/*`, `uv.lock`.
